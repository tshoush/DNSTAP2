The problem we're solvingWhy dnstap, and why out-of-band?

❌ Traditional query logging

  • Runs inside the DNS service path — every query pays the cost.
  • Heavy enough that it's turned on only to chase a problem, then off again.
  • So the data you need is usually not being collected when the incident happens.

✅ dnstap telemetry (DNSTAP2)

  • Out-of-band: the server copies query/response events to a side channel.
  • Negligible impact → safe to leave always on.
  • Feeds Splunk/SIEM (forensics) and Prometheus/Grafana (ops) at once.
Goal: replace reactive, performance-degrading logging with continuous, low-impact visibility into who is asking for what.

DNS server → collectorThe wire: protocol, port & interface

📦 What's on the wire

  • Transport: Frame Streams (fstrm) over TCP.
  • Payload: Protocol-Buffers (dnstap.pb) — query, response, client IP, qname, qtype, rcode, timing, operation (client vs recursive).
  • Direction: NIOS dials into the collector — unidirectional, one stateful firewall rule.

RULE  DNS Server → Collector : TCP 6001

🔌 Which interface?

  • By default NIOS sends all services (incl. DNS) out the LAN1 port.
  • The dedicated MGMT port is off by default; enable it to isolate DNS/telemetry from the data plane.
  • Best practice: keep always-on telemetry off the production path — MGMT or a dedicated LAN2 if available.
No fixed "dnstap port" exists — 6001 is our collector's listener (Vector A/B runs on 6000).

The heart of the pipelineDNS-collector — the receiver

1 · Ingest

Listens on :6001, completes the Frame Streams handshake, reassembles frames.

2 · Decode

Parses the protobuf into structured fields; lowercases qnames; tags client vs recursive (RESOLVER_*).

3 · Route

One ingest, many outputs — a pipeline that forwards each event to all sinks.

Two stores, two questionsMetrics vs. events

📊 Prometheus — "how much / how fast?"

  • Numeric time-series: QPS, rcodes, query types, per-operation rates.
  • Built-in top-N gauges: top domains, requesters, NXDOMAIN, SERVFAIL.
  • Cheap, long-retention, drives alerts.

sum(rate(dnscollector_queries_total[1m]))

🔎 Loki — "which exact events?"

  • Full decoded records, searchable by any field.
  • Answers relational questions Prometheus can't: which domains did recursion resolve?
  • Per-client, per-qname drill-down for investigation.

{job="dnscollector"} | json | dnstap_operation=`RESOLVER_QUERY`

Grafana joins both: counters for the trend, logs for the detail behind any spike.

What you're watching liveThe demo: synthetic traffic

🧪 No NIOS needed to prove the pipeline

  • dnstap_synth.py builds real DNS packets, wraps them in dnstap, frames them over Frame Streams to :6001.
  • Weighted top domains & clients, realistic NXDOMAIN/SERVFAIL mix, ~60% recursive.
  • Exercises the entire chain — collector → metrics → Loki → Grafana.
