D · On the wireFrame Streams + Protocol Buffers

dnstap travels as Frame Streams (a binary, length-framed transport) carrying a protobuf payload. The DNS server dials into the receiver — so it's a single, unidirectional firewall rule.

  • Fields: client IP·port, qname, qtype, rcode, timing, operation (client vs resolver)
  • One rule: DNS member → collector, TCP 6000/6001 inbound. No reverse path.
  • Interface: default LAN1; enable MGMT to keep telemetry off the data plane.
# NIOS-style line the receivers emit <ts> <member> named[id]: client 10.40.150.176#16707 (cdn.fastly.net): query: cdn.fastly.net IN A response: NOERROR # operations carried in dnstap CLIENT_QUERY · CLIENT_RESPONSE ← client-facing RESOLVER_QUERY · RESOLVER_RESPONSE ← recursion leg

K · A real failure, solvedThe Vector accept-queue wedge

Vector was active and listening — but Splunk showed zero Vector events. The accept queue had filled and it stopped accepting connections.

Recv-Q 0 / 128 backlog  accepting ✔
# the smoking gun $ ss -ltnp | grep 6000 LISTEN 129 128 *:6000 users:("vector") ↑Recv-Q ↑backlog → overflow, frames dropped
# the fix $ sudo systemctl restart vector $ ss -ltnp | grep 6000 LISTEN 0 128 *:6000 # drained $ ... dnstap:vector now flowing

L · Built to fit the hostNative binaries, platform-aware

No containers, no Java. Stdlib-only Python orchestration. Installers detect the host and adapt — you don't branch on platform yourself.
