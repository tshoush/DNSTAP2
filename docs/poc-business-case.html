DNS Query Logging vs. dnstap

Should we keep native NIOS query logging, or adopt dnstap for always-on DNS telemetry?
An evidence-based decision — and a working proof it's feasible today.

the problem the approach the cost the proof

Why we're even askingThe problem with native query logging

It taxes the DNS engine

Logging runs inside the resolution path — heavy CPU & I/O. Infoblox's own guidance: it "adversely affects DNS services and performance," and don't enable queries + responses together.

So it gets turned off

Because it's expensive, teams enable it only to chase a problem, then disable it again.

Data is missing when it matters

The query history you need for an incident or threat hunt usually isn't being collected at the moment it happens.
The mandate is to gain continuous DNS visibility without putting DNS availability at risk. That tension is what this POC resolves — with measurements, not opinions.

The decisionOne question, decided by data

Option A — Keep native query logging

  • No new pipeline to run.
  • But pays a performance tax and is realistically on-demand only.
  • Query & response not bundled — weaker for forensics.

Option B — Adopt dnstap (DNSTAP2)

  • Out-of-band, asynchronous — safe to leave always-on.
  • Full query+response fidelity for SIEM / forensics.
  • Cost: a telemetry pipeline + storage to operate.
We will measure both on our own environment and score them against weighted criteria. Primary tie-breaker: whichever better protects DNS availability.

What the evidence already saysVendor & community signal

Infoblox's own guidance

  • Query/response logging "adversely affects DNS services and performance."
  • "Heavy compute demands on the CPU and storage" — can "degrade DNS core… availability."
  • Recommends not running both; offers dnstap (and Data Connector VM) as the low-impact path.

dnstap design & data quality

  • "Fast and lightweight," operates asynchronously, "minimal performance loss."
  • Preserves the complete query+response in binary form — full forensic fidelity.
  • Trade-off acknowledged: a larger storage footprint to plan for.
Strong directional evidence — but we still measure on our hardware and query mix before committing. That's what the test plan is for.

The money questionCost — both options have a hidden bill

Native logging's hidden tax

  • Consumes DDI performance headroom → earlier appliance/member upgrades.
  • Reporting/indexing burden (± Data Connector VM licensing).
  • Incident-response cost: data often not on when needed.

dnstap's visible bill

  • Collector host(s) — small native binaries, optional HA.
  • Storage (Prometheus + Loki) and SIEM ingest volume (size from T8).
  • Day-2 ops effort — but moves load off DNS members (defers upgrades).
3-year TCO compares: appliance/headroom impact · licensing · storage ($/GB × retention × GB/day) · SIEM ingest · FTE. The decision often turns on SIEM ingest vs DDI capacity erosion.

Eyes openRisks & how we address them

Performance regression

Native logging degrading resolution at peak → measure (T2/T4), don't assume.

Backpressure / data loss

Sink loss must never block DNS → prove with T7; characterize buffer/drop.

Storage / SIEM overrun

Cap with retention; size from T8 before committing ingest spend.

Single point of failure

Collector → HA or graceful degradation (T9 sizing).

Privacy / compliance

DNS data is sensitive → retention, access control, encryption — both options.

NIOS build variance

dnstap field names differ → schema discovery handles it; validate on target build.

What's working now

DNS-collector → Prometheus + Loki → Grafana, with Alertmanager rules (receiver-down, NXDOMAIN/SERVFAIL spikes, tunneling heuristic). One-command installers + a synthetic traffic generator for repeatable demos.

Ops-ready

InfoBlox enablement runbook (dry-run → snapshot → apply, per member), firewall/port docs, and an SNMP integration design for host health. Rollout is phased and reversible.
Feasibility is not a question — the pipeline is live. The POC's job is to quantify the trade vs. native logging and decide.

The recommendation pathHow the call gets made

Choose dnstap if…

materially lower DNS impact at peak (T4), DNS stays healthy under sink loss (T7), data needs met (T5/T10), and 3-yr TCO incl. SIEM is acceptable.

Keep native if…

measured peak penalty is negligible and dnstap's storage/SIEM cost is disproportionate — then codify strict on-demand logging guardrails.
