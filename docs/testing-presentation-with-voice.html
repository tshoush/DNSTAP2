The decision we have to makeWhat does dnstap actually add — and can we prove it?

❌ Legacy native query logging

  • Runs inside the DNS service path — Infoblox warns it can "adversely affect DNS performance."
  • So it's turned on to chase an incident, then off again — the data is missing when you need it.
  • One query-only log line; no response, no timing, no recursion view.

✅ dnstap (DNSTAP2)

  • Out-of-band side channel → safe to leave always on.
  • Query and response, correlated, with 60+ fields.
  • The promise: more data, less impact on DNS.
This is a claim. The test exists to prove it on our own hardware — with numbers, not vendor marketing.

Why one server, not twoEliminate hardware variance by construction

✅ Solved for free

  • Same physical box in every mode → no two-server equivalence problem, no cross-over needed.
  • The comparison is apples-to-apples by design.

⚠ The new risk: temporal drift

  • Runs happen at different times → cache warmth or background load could masquerade as a mode difference.
  • Controlled by: consistent warm-up, a fixed run order, and a closing baseline drift check.
The gate: we run the no-logging baseline first and last. If the closing baseline reproduces the opening one (M0' ≈ M0), every delta in between is trustworthy.

The run sequenceFive runs, one mode at a time

RunMode on the memberWhat it tells us
M0All logging OFFThe reference everything is measured against.
M1Native query logging onlyThe "normal" legacy logging penalty.
M2Native query + responseThe heavy mode Infoblox advises against — quantified.
M3dnstap → collectorThe proposed always-on path.
M0'All logging OFF againDrift check — should reproduce M0.
Only one mode is enabled at a time; the previous one is disabled before the next is turned on — so each run measures exactly one logging path.

The trap, and how we avoid itThe cold-cache confound

Every GUI mode switch restarts DNS and flushes the resolver cache. Start load too soon and you measure upstream latency, not the logging mode.

Preferred — test authoritatively

  • Point the query file at zones the member is authoritative for.
  • No cache, no upstream, no warm-up — the confound simply disappears.
  • Isolates exactly the cost we care about: serve + log.

If testing the recursive path

  • Settle after restart → identical warm-up → measure steady-state only.
  • Make cache state identical at the start of every run.
  • CPU & disk are cache-independent; warm-up protects latency & QPS.

Member CPU/mem are read with built-in OIDs — CPU 1.3.6.1.4.1.7779.3.1.1.2.1.8.1.1.0, memory …8.2.1.0. NIOS doesn't surface disk I/O over SNMP, so the native-logging disk-write cost is read from the member's stats.

The testsFour questions, asked of every mode

IDTestWhat it answers
T2Max QPS ceiling (ramp)How high can the member go before the SLA breaks — per mode?
T3Fixed-load latency / CPU / memAt steady load, what does each mode cost in latency and resources?
T4Soak (2–4 h)Any drift, leak, or log-disk fill over time?
T7Failure / safetydnstap: kill the collector — does DNS stay healthy? native: disk/IO pressure.
T7 is the safety proof: losing the telemetry sink must never threaten resolution — that's the whole point of out-of-band.

What comes outThe comparison the decision needs

The headline tables

  • Drift check: M0 vs M0' — the trust gate.
  • QPS ceiling per mode — and the % each logging mode costs vs baseline.
  • Latency / CPU / memory at steady load, side by side.
  • Storage — bytes/event → GB/day at retention.

Already known from the lab (to be confirmed on the member)

  • Collector cost: ~200 µs CPU/event, ~205 MB RAM.
  • Storage: 1,745 B/event raw → 54.7× gzip.
  • Fidelity: 60+ fields vs one query-only log line.
The pipeline's cost is already shown small; this test supplies the missing, decisive piece — the DNS-server penalty of each mode.

How we decideGo / no-go framework

Recommend dnstap if…

  • It shows materially lower DNS impact than native logging at peak (T2/T3).
  • DNS stays healthy when the collector is lost (T7).
  • Storage / SIEM cost is acceptable (T3 bytes/event + sizing).

Stay on native logging only if…

  • Its measured penalty is negligible at our real peak, and
  • dnstap's storage/SIEM cost is disproportionate to the value.
  • Then: codify strict on-demand logging with guardrails.
A tie breaks toward whichever better protects DNS availability — the primary mandate.
